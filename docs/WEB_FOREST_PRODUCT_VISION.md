# Token Forest 网页个人森林与桌面联动产品方案

> 文档状态：产品讨论稿 v0.1  
> 日期：2026-07-13  
> 适用范围：Token Forest 网页个人森林、桌面联动、后续森林经营与温和社交  
> 当前阶段：产品分析与方向验证，不代表实现承诺  
> 推荐流程：讨论与拍板 → 交互原型 → 用户验证 → 技术设计 → 代码实施

## 0. 执行摘要

### 0.1 核心判断

Token Forest 的网页森林不应只是官网新增的一张展示页，也不应只是把排行榜里的多树弹窗放大。它应该逐步成为产品长期体验的主场：

> **桌面端是工作时陪着用户的那棵树，网页端是这棵树生活的整片森林。**

桌面与网页不是两套独立农场，而是同一个生命、两种观察尺度：

| 端 | 核心职责 | 典型使用时刻 |
|---|---|---|
| 桌面端 | 本地读取、即时反馈、低打扰陪伴 | 用户正在使用 Claude Code / Codex 工作 |
| 网页端 | 森林全景、回顾、探索、布置、分享 | 用户主动回访自己的长期成果 |
| 云端 | 用户明确授权后的身份与私有桥梁 | 设备配对、状态同步、跨端事件转发 |
| 移动网页 | 查看、拍照、分享、轻互动 | 用户离开工作电脑时快速访问 |

产品最终希望带来的感受不是“我又完成了一项任务”，而是：

> **这是我用真实工作与创作时间养出来的森林。**

### 0.2 推荐首版方向

推荐以两个阶段共同构成第一次正式的网页森林体验：

1. **M0 · 私人 Forest Gallery**：用户完成登录与设备配对，在网页看到由桌面真实状态生成的私人森林。
2. **M1 · Living Link**：桌面收取 token 气泡后，网页森林实时产生变化；网页抚摸树后，桌面树轻轻回应。

第一次发布最重要的不是商店、任务或好友系统，而是证明：

> **用户在两端看到的是同一棵树。**

### 0.3 首版明确不做

- 不复制完整 QQ 农场经营系统。
- 不做实时多人同屏。
- 不做偷果实、破坏或竞争性资源争夺。
- 不因长期未登录、未浇水或错过收获而惩罚用户。
- 不通过更高 token 消耗直接发放更高价值的稀有奖励。
- 不在首版开放网页收果、余额修改、购买或复杂布置等耐久写操作。
- 不用网页端取代桌面端的本地日志读取职责。
- 不未经授权上传 prompt、回答正文、源代码、项目名、会话标题或本地路径。
- 不在首版重做全部美术为等距 3D 或完整 3D 世界。

## 1. 背景与产品机会

### 1.1 Token Forest 当前解决的问题

程序员每天使用 Claude Code、Codex 等 AI 工具时，会产生大量 token。传统用量工具通常把它们表现为数字、表格、曲线和费用，但这些信息缺少情感重量，也很难让用户感知“今天到底完成了多少事情”。

Token Forest 当前通过桌面像素树完成了第一步转化：

1. 本地读取 Claude Code / Codex 用量日志。
2. 将新增 token 表现为桌面气泡。
3. 用户收取气泡，树获得成长。
4. 树从种子经历 8 个阶段，逐渐成熟。
5. 多棵树、果实、装饰和成绩单保存长期使用痕迹。

它解决的不只是“怎么看 token”，而是：

> **如何把抽象消耗转化为可见、可养、可以产生感情的东西。**

### 1.2 为什么网页森林是自然的下一步

桌面树适合陪伴工作，却天然受限于尺寸和注意力预算：

- 它必须保持小、轻、安静。
- 它不适合承载完整世界、地图和复杂布置。
- 它更适合展示当下，而不是回顾数月或数年的积累。
- 多树、动物、地貌和长期历史在桌面浮窗中难以展开。

网页端则适合承担更大的空间尺度：

- 展示所有树和完整生态。
- 通过镜头、路径和场景建立“这是一个地方”的感觉。
- 回顾树的出生、阶段变化和重要里程碑。
- 提供拍照、分享和后续布置能力。
- 在不干扰工作的情况下，允许用户偶尔沉浸几分钟。

因此网页森林不是桌面功能的重复，而是桌面体验的自然展开。

### 1.3 独特市场位置

Token Forest 位于四类产品的交叉点：

1. 将真实行为映射为成长的效率工具。
2. 通过长期积累形成情感资产的养成产品。
3. 低打扰地停留在桌面的陪伴型应用。
4. 支持异步访问、展示与轻互动的个人空间。

它不是纯 token tracker、传统电子宠物、完整农场模拟，也不是简单的数据看板。其独特性在于：

> **用户真实使用 AI 的行为，会在一个平静、长期存在的数字世界中留下生命痕迹。**

## 2. 当前产品与网站审计

### 2.1 当前网站定位

截至本文撰写时，TT-website 已经是一个完成度较高的：

> **营销官网 + 产品演示 + 公共排行榜**

现有主导航包含：

- 首页
- 下载
- 数据面板
- 排行榜
- GitHub

但尚无普通玩家使用的：

- 我的森林
- 网页账号
- 设备配对
- 私人森林设置
- 访客或分享管理

Ranger 是排行榜管理员入口，不是玩家账户系统。

### 2.2 已有能力

#### 多树真实数据

桌面端已经可以向公开排行榜同步：

- 总 token。
- 当前主树。
- 当前主树阶段。
- 地区。
- 每个树种自己的 token 与阶段。

网站会将主树排在第一位，并在排行榜弹窗中横向展示一个用户的多棵树。相关证据：

- [排行榜数据归一化](../lib/leaderboard.ts)
- [多树弹窗](../components/tree-modal.tsx)
- [每树快照迁移](../supabase/migrations/0004_leaderboard_trees.sql)

#### 四树种 × 八阶段

当前已有：

| 树种 | 环境主题 | 成熟阈值 |
|---|---|---:|
| 苹果树 | 白天、云、远山、果园 | 90M |
| 樱花树 | 晨曦、山、雾、花瓣、神社感 | 270M |
| 仙人掌 | 西部落日、台地、风沙 | 360M |
| 圣诞树 | 冬夜、月亮、星星、落雪 | 450M |

每个树种都具备 8 个成长阶段，网页展示使用的不是随意演示阈值，而是与桌面产品一致的成长体系。

#### 丰富的视觉与交互原语

现有网站已经具备：

- token 气泡自动出现。
- 点击收取、爆裂碎屑和浮字。
- 树呼吸、成长抖动和阶段渐变。
- 长按让树成长。
- 树种切换和多树画廊。
- 花瓣、风沙、滚草、落雪、星空、萤火虫和流星。
- Dashboard 三页签交互。
- 宝箱逐帧动画。
- reduced-motion 与离屏暂停。

主要位置：

- [单树场景](../components/tree-scene.tsx)
- [树种 Showcase](../components/tree-showcase.tsx)
- [token 气泡](../components/token-bubbles.tsx)
- [全局动画与设计变量](../app/globals.css)

#### 尚未充分使用的环境资产

现有 sprites 目录中还有：

- 竹篱笆、普通篱笆和破损篱笆。
- 篮子。
- 石灯笼。
- 秋千。
- 鸟居。
- 风铃。
- 宝箱。
- 各树种的主题装饰。

这些可以直接成为第一版森林的地块边界、交互点和场景标志物。资产见 [public/sprites](../public/sprites)。

### 2.3 当前联动的真实边界

当前桌面到网站的链路是：

~~~mermaid
flowchart LR
    A["桌面本地日志"] --> B["桌面树与本地存档"]
    B -->|"用户主动开启排行榜"| C["Supabase 匿名身份"]
    C --> D["覆盖上传一行排行榜快照"]
    D --> E["网站公共排行榜"]
    E --> F["多树弹窗"]
~~~

当前事实：

- 排行榜默认关闭。
- 首次开启会展示字段同意确认。
- 开启后约每 30 分钟同步。
- 换树、恢复开启、修改昵称或地区、退出时也可能触发上传。
- 上传为当前整份快照，按匿名 user_id 覆盖同一行。
- 网站只能读取，不能向桌面发送动作。
- 没有私人森林、设备配对或实时双向联动。

桌面同步说明见 [Token-Forest 排行榜文档](../../Token-Forest/docs/LEADERBOARD.md)。

### 2.4 当前身份不是可恢复的玩家账号

现有排行榜使用 Supabase 匿名认证：

- 首次实际上传时创建匿名身份。
- user_id、access token 和 refresh token 保存在单台设备本地。
- 它用于保证客户端只能修改自己的排行榜行。
- 普通用户不能用邮箱或 GitHub 登录该身份。
- 当前没有跨设备恢复和设备配对体验。

因此不能把现有匿名排行榜身份直接称为“Token Forest 账号”，也不能假设它天然支持网页个人森林。Supabase 也明确说明匿名用户在换设备或清除本地数据后无法自然恢复原匿名身份，见 [Supabase Anonymous Sign-ins](https://supabase.com/docs/guides/auth/auth-anonymous)。

### 2.5 当前数据模型不足以承载森林世界

现有 trees JSONB 形态类似：

~~~json
{
  "apple": {"tokens": 90000000, "stage": 7},
  "cherry": {"tokens": 12000000, "stage": 3}
}
~~~

它适合排行榜里的四树概览，但无法表达：

- 两棵苹果树。
- 每棵树独立的 tree_id。
- 树名、出生时间和成长故事。
- 森林坐标、朝向、遮挡层级和地块。
- 果实、背包和装饰实例。
- 浇水、抚摸、冷却或最后互动状态。
- 网页与桌面之间的领域事件。

正式森林需要将“树实例”与“树种”分离。

### 2.6 Dashboard 页面不能被误认为个人网页数据面板

当前网页数据面板是桌面功能的精致营销演示：

- 三页签自动轮播。
- 数字为固定虚构示例。
- 不读取当前访问者的真实桌面数据。

因此个人森林不能只是在 Dashboard 上添加一块树景，而需要独立的登录后产品域。

### 2.7 当前版本与目标版本

| 维度 | 当前 | 网页森林目标 |
|---|---|---|
| 身份 | 单设备匿名排行榜会话 | 可恢复用户账号 + 可撤销设备 |
| 可见性 | 公共排行榜 | 私人默认，分享与排行分别授权 |
| 数据 | 每树种聚合快照 | 独立树、生态、布局、事件与版本 |
| 方向 | 桌面单向上传 | 桌面成长上行 + 网页瞬时互动下行 |
| 频率 | 低频快照 | 快照 + 幂等事件 + 实时通知 |
| 页面 | 营销和榜单 | 长期存在的个人森林 |
| 隐私 | 排行榜同意字段 | Forest Sync 独立同意和数据管理 |

## 3. 竞品与相邻产品研究

### 3.1 研究结论

目前没有一个现成产品同时做好：

- 真实 token 行为映射。
- 漂亮可交互的森林世界。
- 低打扰桌面陪伴。
- 网页与桌面双向联动。
- local-first 与用户可控公开。

最适合 Token Forest 的组合不是照搬某一个产品，而是：

> **Forest 的行为意义映射 × GitAnimals 的开发活动自动映射 × Rusty’s Retirement 的桌面陪伴 × Animal Crossing 的场所依恋 × QQ 农场的异步来访 × Cozy Grove / Viridi / Kinder World 的温和节奏。**

### 3.2 竞品总览

| 产品 | 已验证的核心机制 | Token Forest 可借鉴 | 明确避坑 |
|---|---|---|---|
| QQ 农场／开心农场 | 持久土地、等待、收获、异步好友访问 | 私人领地、离线变化、访客痕迹、短时回访 | 偷资源、准点上线、过期损失、多货币堆叠 |
| Forest | 现实专注变成永久森林记录 | 让工作历史成为可回看的景观 | 离开就死树、内疚型惩罚 |
| GitAnimals | GitHub 活跃自动养宠并嵌入个人主页 | 开发行为自动映射、身份展示、分享卡片 | 刷指标、随机抽取、交易主导 |
| Rusty’s Retirement | 农场安静停留在屏幕边缘 | 桌面低打扰、几秒内完成互动 | 高频点击经济、复杂桌面经营 |
| Spirit City | 氛围空间、专注工具、居民与装饰 | 光线、声音、环境居民和收藏惊喜 | 再造一整套待办工具 |
| Animal Crossing | 现实时间、季节、布置、居民和来访 | 场所依恋、稳定空间、长期记忆 | 首版复制完整生活模拟 |
| Cozy Grove | 温和回访、自然结束点、异步礼物 | 离线摘要、每日几分钟、善意社交 | 固定任务清单、积压和 FOMO |
| Focus Plant | 行为产出可分配给不同植物 | 用户选择养哪棵树或哪个林区 | 满屏入口、能量限制、付费加速 |
| Viridi | 缓慢生长、第二窗口陪伴 | 安静、亲密、无经济收益的触摸 | 缺席导致死亡、交互过少 |
| Kinder World | 无评判的短时照料 | 温和文案、植物不会责怪用户 | 强行包装成心理治疗产品 |

### 3.3 QQ 农场／开心农场

真正有效的并不只是“种菜”，而是：

- **所有权**：这是我的地。
- **期待感**：我离开后，这里仍会变化。
- **社交痕迹**：别人来过，并留下可见结果。

适合借鉴：

- 长期存在的私人林地。
- 离线期间的变化摘要。
- 好友异步来访，而非要求同时在线。
- 阳光、脚印、明信片、小礼物等善意痕迹。
- 一次只需要几分钟的回访。

不应照搬：

- 偷果实。
- 资源被夺走。
- 必须准时收获。
- 枯萎、腐烂和连续签到。
- 矿山、装备、订单、礼包和多货币无限堆叠。

参考：[QQ 农场 App Store](https://apps.apple.com/cn/app/qq%E5%86%9C%E5%9C%BA/id506655146)、[QQ 经典农场 2026 回归报道](https://tech.cnr.cn/techph/20260206/t20260206_527518544.shtml)。

### 3.4 Forest

Forest 的关键不是树木美术，而是建立了极清楚的意义映射：

> 我投入的现实时间，不只是一条统计，而是森林里永久存在的一棵树。

适合借鉴：

- 真实行为必须在世界中留下永久证据。
- 森林同时承担成果展示、历史记录和情绪奖励。
- 支持本周、本月、今年等长期全景。
- 生成可分享的森林卡片。

不应照搬：

- 中断就死亡的惩罚。
- 用单一数量判断用户是否努力。
- 让用户为奖励主动制造无意义消耗。

参考：[Forest App Store](https://apps.apple.com/us/app/forest-focus-for-productivity/id866450515)。

### 3.5 GitAnimals

GitAnimals 与 Token Forest 的目标用户和数据来源非常接近。它证明：

> 最好的开发者养成产品，不要求用户为了养成而改变原有工作流程。

适合借鉴：

- 用户本来就在发生的开发活动自动推动成长。
- 结果可以成为开发者身份的一部分。
- 提供只读分享页、README 卡片或个人网站嵌入。

不应照搬：

- 按原始 token 数随机抽取大量稀有内容。
- 交易和市场成为主要动力。
- 公开排名成为默认目标。

参考：[GitAnimals GitHub](https://github.com/git-goods/gitanimals)。

### 3.6 Rusty’s Retirement 与 Spirit City

Rusty’s Retirement 验证了：

> 陪伴感来自持续可见，但持续可见不应等于持续索取注意力。

Spirit City 则说明效率产品可以拥有强空间感、光线、声音、居民和收藏惊喜。

Token Forest 应：

- 将复杂浏览和布置放到网页。
- 让桌面保持小、轻、安静。
- 用微小环境变化表达长期进展。
- 让动物或精灵因工作习惯和环境组合而出现。

不应：

- 在桌面浮窗堆叠任务、商店和背包。
- 重新建设番茄钟、待办、日历和笔记系统。
- 让动画或通知遮挡编辑器和终端。

参考：[Rusty’s Retirement](https://store.steampowered.com/app/2666510/Rustys_Retirement/)、[Spirit City](https://store.steampowered.com/app/2113850/Spirit_City_Lofi_Sessions/)。

### 3.7 Animal Crossing 与 Cozy Grove

Animal Crossing 的长期价值来自“场所依恋”：

- 用户记得一棵树种在哪里。
- 记得某个角落怎样逐渐布置完成。
- 记得某个季节第一次出现的景色。
- 即使没有任务，也愿意回去看一眼。

Cozy Grove 的重要启发是“自然结束点”：用户完成几分钟温和互动后，可以安心离开。

Token Forest 应：

- 让森林拥有稳定、可辨认的区域和路径。
- 使用昼夜、季节、天气和居民建立时间感。
- 让一次回访在 30 秒到 3 分钟内也能完整结束。
- 采用明信片、小礼物和脚印等异步社交。

不应：

- 首版构建大量 NPC、家具和故事内容。
- 让用户每天面对任务债务。
- 用限时活动制造错失焦虑。

参考：[Animal Crossing](https://my.nintendo.com/rewards/3d951ffe994d5193)、[Cozy Grove: Camp Spirit](https://www.netflix.com/tudum/articles/cozy-grove-camp-spirit-release-date-news)。

### 3.8 Viridi 与 Kinder World

Viridi 证明数字植物不需要复杂任务也能产生陪伴价值。Kinder World 的启发则是：

> 与植物互动可以只是表达关心，不一定要换取资源。

Token Forest 应：

- 让抚摸、吹风和轻敲树干不产生经济收益。
- 使用温和、无评判的回归文案。
- 长期不来，树也不会死亡或责怪用户。
- 让不同树种具有不同的互动性格。

参考：[Viridi](https://store.steampowered.com/app/375950/Viridi/)、[Kinder World](https://www.playkinderworld.com/game)。

## 4. 产品定位与设计原则

### 4.1 一句话定位

> **Token Forest 是一个把真实 AI 使用与创作过程，转化为可成长、可照料、可回访的个人数字森林的陪伴型产品。**

### 4.2 目标用户

#### A. 长期 AI 工具使用者

- 经常使用 Claude Code、Codex 等工具。
- 对 token 数据有兴趣，但不满足于图表。
- 希望看到长期积累，而不是每日归零的统计。

#### B. 喜欢桌面陪伴体验的用户

- 喜欢桌面宠物、像素画、放置和安静的环境动画。
- 不一定关心复杂 token 指标。
- 更在意树是否可爱、是否有生命感、是否记得自己。

#### C. 收藏与布置型用户

- 喜欢解锁植物、动物、环境和装饰。
- 愿意偶尔回访，但不希望承担高频任务。
- 有分享森林明信片或展示个人空间的需求。

#### D. 轻社交与身份展示用户

- 希望将自己的 AI 创作历史变成一种身份表达。
- 可能参与排行榜，但更关心森林是否独特。
- 愿意让朋友只读来访并留下善意痕迹。

### 4.3 用户任务

功能任务：

- 快速理解最近的工作让森林发生了什么。
- 在网页看到所有树和长期成长。
- 回顾树的阶段、年龄和重要里程碑。
- 在多个设备上访问同一私人森林。
- 拍摄和分享森林，而不暴露敏感工作信息。

情感任务：

- 觉得“这棵树是我养出来的”。
- 让工作消耗留下可爱的、长期存在的东西。
- 在高强度工作之间获得短暂放松。
- 离开一段时间后仍愿意回来。

信任任务：

- 知道哪些数据仍在本地。
- 知道哪些字段被同步。
- 能区分私人同步、公开分享和排行榜。
- 能随时暂停同步、撤销设备和删除云端森林。

### 4.4 产品设计原则

#### 原则 1：同一个生命，两种尺度

桌面和网页不能像两个独立游戏。两端必须共享同一棵树、同一阶段、同一成长历史和清晰的最后同步状态。

#### 原则 2：工作产生生命，不制造额外工作

森林是用户正常工作后自然形成的结果，不应再要求用户完成浇水、清任务、连续签到等家务。

#### 原则 3：树不会死，只会休息

- 不因长期未登录死亡。
- 不因漏掉照料降级。
- 不因错过时间窗损失果实。
- 回归用户不会面对任务债务。

#### 原则 4：短暂访问也能完整结束

一次网页访问可以只有 30 秒到 3 分钟：

1. 看离线变化。
2. 摸摸树或点亮一盏灯。
3. 拍一张森林明信片。
4. 安心离开。

#### 原则 5：使用真实数据，但不鼓励浪费 token

token 是历史事实，不是需要刷取的积分。高 token 使用可以被记录，但高价值奖励不能无限线性增长。

#### 原则 6：默认私密，公开分别授权

私人 Forest Sync、公开排行榜、公开分享链接必须是三个独立选择。

#### 原则 7：场景优先于报表

网页森林首先应让用户觉得“回到了一个地方”，树的统计作为可展开档案存在，而不是把 Dashboard 放大成首屏。

#### 原则 8：异步善意优先于实时多人

优先做只读来访、脚印、明信片和阳光；实时多人、聊天和复杂公会暂缓。

#### 原则 9：一个清楚的事实源

首期由一台桌面设备作为耐久成长的主要写入者。网页只处理不影响经济的瞬时互动，避免两端都能改余额和成长造成冲突。

#### 原则 10：70% 安静陪伴，30% 轻量经营

产品应有足够互动与长期目标，但不能变成高频农场任务系统。

## 5. 森林世界的核心模型

### 5.1 为什么不应按每次 token 生成一棵树

如果每次会话、每个气泡或每一定量 token 都生成一棵完整树，会产生：

- 森林快速拥挤。
- 单棵树失去情感身份。
- 用户难以理解哪棵树代表什么。
- 视觉逐渐退化为“漂亮的数据点排列”。
- 大量树实例带来布局与性能成本。

因此推荐采用三层模型。

### 5.2 第一层：四棵主树

首版继续保留四棵主树，每棵是长期角色，而非一次性记录。

未来每棵主树应拥有：

- 唯一 tree_id。
- 树种。
- 自定义名称。
- 出生时间。
- 当前成长值与阶段。
- 历史累计 token。
- 最近成长时间。
- 所在林区。
- 已装备装饰。
- 关联的成长事件。

主树是用户最主要的情感对象，也是桌面与网页之间的同一实体。

### 5.3 第二层：森林生态

主树成长同时推动环境丰富：

- 草地、花朵和灌木密度。
- 路径、桥、水域和石头。
- 灯光、风铃和小型建筑。
- 鸟、蝴蝶、萤火虫和其他居民。
- 天气和季节表现。
- 可放置装饰区域。

这样即使只有四棵主树，也能形成一片真正丰富的森林。

### 5.4 第三层：记忆树

后续可以在以下时刻生成记忆树：

- 主树第一次成熟。
- 一个月、一年或版本里程碑。
- 用户主动将一段成长“移栽”为永久纪念。
- 完成一个用户主动命名的创作时期。

记忆树不再承担实时 token 增长，而是保存一段历史。它可以记录：

- 时间范围。
- 当时的树种和阶段。
- 累计成长值。
- 用户主动填写的标题或一句话。

默认不自动上传项目名称或会话内容。

### 5.5 四类林区

| 林区 | 气质 | 现有可复用资产 | 主要互动 |
|---|---|---|---|
| 苹果果园 | 温暖、丰收、木质感 | 秋千、篮子、木篱笆 | 摇果、秋千、鸟、落叶 |
| 樱花神社 | 宁静、仪式感、庭院 | 鸟居、石灯、风铃、竹篱 | 花瓣、风铃、灯光、小溪 |
| 仙人掌峡谷 | 干燥、开阔、荒野 | 破篱笆、矿洞感装饰、路牌 | 强风、滚草、日落、沙尘 |
| 冬日林地 | 节日、雪夜、暖灯 | 雪人、礼物、灯串、雪篱 | 落雪、脚印、灯光、礼物 |

### 5.6 成长与经济的四层含义

| 数值 | 作用 | 是否可消费 | 原则 |
|---|---|---:|---|
| 原始 token | 真实历史记录 | 否 | 永远保留，不因消费减少 |
| 成长值 | 驱动树阶段 | 否 | 可对极高用量采用递减收益 |
| 生态成熟度 | 驱动环境丰富 | 否 | 更看长期与多样性，不看单日峰值 |
| 果实／装饰货币 | 购买少量装饰 | 是 | 来自里程碑与稳定成长，不鼓励刷 token |

首版不急于写死换算公式，但应遵守：

- 低到中等使用能获得明确成长。
- 极高使用仍被记录，但游戏奖励边际收益逐渐降低。
- 长期稳定使用比单日突发消耗更容易形成丰富生态。
- 不让树停止记录真实 token。

## 6. 核心循环与反馈节奏

### 6.1 总体闭环

~~~text
使用 AI 工作
→ 桌面端检测 token
→ 生成并收取成长气泡
→ 主树获得成长
→ 网页森林产生可见变化
→ 用户回访、照料、布置或分享
→ 带着新的情感连接返回工作
~~~

### 6.2 即时循环

时长：数秒。

1. Claude Code / Codex 产生新的使用量。
2. 桌面出现气泡。
3. 用户点击收取。
4. 桌面树立即发光、抖动或吸收能量。
5. 若网页在线，森林出现对应颜色的萤火能量。
6. 能量进入同一棵树，树冠产生环境反馈。

### 6.3 每日／每周循环

时长：30 秒到 3 分钟。

1. 查看离开期间发生的变化。
2. 聚焦最近成长的树。
3. 完成一次轻互动。
4. 看到新出现的花、动物或小装饰。
5. 可选拍摄明信片。
6. 自然离开。

不设置“今日必须完成”的固定清单。

### 6.4 长期循环

时长：数周到数年。

- 主树跨越成长阶段。
- 林区逐渐恢复或丰富。
- 解锁新的居民与环境变化。
- 形成记忆树。
- 逐渐完成个性布置。
- 森林成为用户 AI 创作历史的长期容器。

### 6.5 输入到反馈的映射

| 输入事件 | 桌面反馈 | 网页反馈 | 是否耐久 |
|---|---|---|---:|
| AI 开始产生 token | 气泡、胶囊灯环 | 树冠微弱呼吸、远处萤火 | 否 |
| 用户收取气泡 | 吸收、浮字、抖动 | 能量沿路径进入树 | 是 |
| 阶段升级 | 强成长动画 | 林区光线变化、生态解锁 | 是 |
| 果实生成 | 果实出现 | 果园或林区产生丰收提示 | 是 |
| 网页抚摸树 | 网页树冠局部波动 | 桌面树轻摇或落叶 | 否 |
| 网页吹风 | 花瓣、叶子、雪飘动 | 桌面短暂风效 | 否 |
| 用户长时间离开 | 无惩罚 | 回归时展示变化摘要 | 是 |

## 8. 视觉与空间方向

### 8.1 推荐：横向 2.5D 像素森林

现有树木为正面透明 PNG，画布尺寸与比例差异较大。最适合的方向是：

- 保留正面树木。
- 使用前景、中景、后景和遮挡营造深度。
- 横向拖动或滑动穿行四个林区。
- 使用视差、雾、光线和环境粒子增强空间。
- 对每棵树定义统一的“树脚锚点”、占地和点击区域。

不建议首版采用完整等距或 3D：

- 现有资产复用率低。
- 需要重绘多个方向和大量地形。
- 交互与碰撞复杂度上升。
- 移动端和低性能设备风险更高。

### 8.2 场景层级

由后向前：

1. 天空、远山和天气。
2. 远景森林剪影。
3. 地貌、水域和路径。
4. 主树与大型装饰。
5. 灌木、花朵和动物。
6. 前景枝叶、雾和环境粒子。
7. 交互提示与 HUD。

### 8.3 镜头原则

- 首次进入聚焦最近成长的树。
- 鼠标拖动或触摸滑动平移。
- 滚轮或双指缩放。
- 双击树快速聚焦。
- 始终提供“回到主树”。
- 限制镜头边界和缩放范围。
- 拍照模式可隐藏 UI。
- 移动端切换林区可使用吸附分页，避免迷失。

### 8.4 昼夜、天气与季节

逐步加入：

- 按用户本地时间变化的昼夜。
- 与林区相符的局部天气。
- 柔和雨、雪、风和雾。
- 里程碑时出现彩虹、流星或萤火虫群。
- 用户可选择固定到喜欢的氛围用于拍照。

时间与天气主要用于生命感，不应改变经济收益。

### 8.5 声音

- 默认关闭，或首次明确询问。
- 环境音与互动音分开控制。
- 风铃、水声、风、雪等保持低音量。
- 桌面端继续服从安静、胶囊和专注模式。
- 分享页默认静音。

### 8.6 无障碍

- 支持 prefers-reduced-motion。
- 关键交互不能只依赖悬停。
- 所有可交互对象提供键盘焦点。
- 不用颜色作为唯一状态表达。
- Claude 橙和 Codex 蓝同时配合图形或文字。
- 移动端使用底部信息卡，不依赖右键。
- 粒子、视差和前景遮挡均允许关闭。

## 9. 交互设计

### 9.1 交互层次

森林互动分为五类，按风险和实现顺序逐步开放：

| 层次 | 目的 | 例子 | 是否影响耐久状态 | 建议阶段 |
|---|---|---|---:|---|
| 环境反馈 | 让世界看起来活着 | 风、光、动物、树呼吸 | 否 | M0 |
| 查看与聚焦 | 理解树和历史 | 点击档案、聚焦、成长进度 | 只读 | M0 |
| 情感触摸 | 建立“同一生命”感 | 抚摸、吹风、轻敲 | 否，短 TTL | M1 |
| 经营操作 | 建立所有权和个性 | 收果、摆装饰、点灯 | 是 | M2 |
| 社交互动 | 留下善意痕迹 | 阳光、明信片、脚印 | 部分 | M3 |

### 9.2 树木互动

#### 悬停树冠

- 叶片在指针附近轻轻晃动。
- 不弹出大面积 Tooltip。
- 仅显示树名、阶段和一个明显的“查看”暗示。
- 减少动画模式只显示轮廓高亮。

#### 点击树

打开紧凑的树木档案：

- 树名和树种。
- 出生日期和种植天数。
- 当前阶段。
- 累计成长值和 token。
- 最近一次成长。
- 距下一阶段。
- 最近解锁的生态变化。

精确模型、会话和项目统计仍留在桌面 Dashboard，不因 Forest Sync 自动上传。

#### 抚摸

- 指针经过树冠产生局部波动。
- 不增加成长值、不掉落货币。
- 若桌面在线，发送短期 tree_petted 事件。
- 桌面树以轻摇、落叶或小光点回应。
- 高频连续触发需要合并和限流，避免桌面持续抖动。

#### 吹风

- 用户按住并拖动，或点击风图标。
- 苹果树落叶、樱花飘瓣、仙人掌扬尘、圣诞树落雪。
- 事件仅用于氛围，允许丢失，不进入永久账本。

#### 轻敲树干

- 产生很短的音效。
- 可能惊起鸟或晃动挂件。
- 不直接掉落果实，避免形成高频点击劳动。

#### 长按

- 展示成长能量沿树干流动。
- 显示下一阶段的可读进度。
- 不隐藏在长按里的独占信息，移动端和键盘仍有替代入口。

### 9.3 环境互动

- 点击风铃：播放一次轻声并摇动。
- 点击秋千：摆动数秒。
- 点击石灯：切换本地预览灯光；M2 才保存状态。
- 点击水面：产生涟漪。
- 点击雪地：留下短暂脚印。
- 鸟受惊飞走后自然返回。
- 宝箱：M0 可作为纯动画彩蛋；M2 才连接收藏系统。
- 篮子：M2 用于果实或装饰入口。

### 9.4 离开期间变化摘要

用户回访时先展示一张可跳过的摘要，而不是日志洪流：

> 欢迎回来。你离开的 5 天里：
>
> - 苹果树进入了第 VII 阶段。
> - 樱花林多了两盏石灯。
> - 一只新鸟开始在清晨来访。

原则：

- 强调发生了什么，不强调错过了什么。
- 不出现“未领取”“已过期”“连续中断”等负面提示。
- 用户可一键进入最近变化位置。
- 若离线或同步暂停，明确说明摘要基于哪个时间点的状态。

### 9.5 拍照与分享

拍照模式应尽早加入，因为漂亮森林本身就是传播媒介：

- 隐藏 HUD。
- 调整有限镜头。
- 可选择显示或隐藏树名、阶段和 token。
- 可选择固定昼夜和天气，仅影响照片。
- 生成图片和只读分享链接。
- 分享页默认不显示精确 token、在线状态和内部用户 ID。
- 分享链接可撤销、重新生成或设置过期时间，后续评估。

## 10. 关键用户旅程

### 10.1 第一次使用桌面端

1. 用户无需注册即可在本地养树。
2. 桌面读取本地 token 记录。
3. 用户看到第一个成长气泡。
4. 用户收取后，树产生第一次明确成长。
5. 在用户已理解本地价值以后，桌面出现轻入口：

   > 在网页看看它生活的森林

6. 不在首次启动时阻断用户并强迫注册。

目标：先建立感情和价值，再提出账号与同步需求。

### 10.2 首次打开网页森林

1. 用户从桌面菜单选择“打开我的森林”。
2. 浏览器进入登录与设备配对页。
3. 用户通过 Email Magic Link、OTP 或后续 GitHub OAuth 登录。
4. 桌面显示一次性短期配对码。
5. 网页说明：
   - 将同步哪些数据。
   - 哪些数据永远留在本地。
   - 私人同步不会自动加入排行榜。
6. 用户确认后，网页加载真实森林快照。
7. 镜头从雾或剪影中落到最近成长的树。
8. 显示：

   > 你的苹果树从桌面抵达了森林。

9. 用户可立即点击树查看档案。

成功条件：用户在开始配对后一分钟内看到并认出自己的树。

### 10.3 两端联动的魔法时刻

1. 用户同时打开桌面树与网页森林。
2. 桌面产生一枚 Claude 或 Codex 气泡。
3. 用户在桌面点击收取。
4. 桌面立即完成本地成长。
5. 网页出现对应来源的萤火能量。
6. 能量进入同一棵树，树冠发光或长出新芽。
7. 用户在网页抚摸这棵树。
8. 桌面上的树在目标延迟内轻摇、落叶或出现光点。

这是 M1 最关键的体验验证。

### 10.4 长时间离开后回归

1. 用户重新打开网页。
2. 页面先显示森林状态时间：

   > 显示的是 12 分钟前同步的状态

3. 若有变化，展示温和摘要。
4. 用户可逐项查看，也可直接进入森林。
5. 不显示损失、积压任务或责备文案。
6. 如果桌面仍离线，网页不会假装实时。

### 10.5 离线工作与重新连接

1. 桌面断网后仍照常读取本地日志和养树。
2. 私人 Forest Sync 事件进入本地 outbox。
3. 网页继续显示最后确认的快照，并显著标注时间。
4. 桌面恢复联网后：
   - 先确认账号和设备仍有效。
   - 使用原事件 ID 重试未确认事件。
   - 拉取服务端最新 revision。
   - 上传缺失事件或快照。
5. 网页显示：

   > 已补同步 3 次成长

6. 不突然回滚用户刚看到的树。

### 10.6 解除设备、暂停与删除

用户必须能区分：

- 暂停同步：本地继续养树，云端停在最后快照。
- 解除此设备：撤销设备凭据，不影响本地存档。
- 删除云端森林：执行可重试、可确认的删除任务。
- 退出排行榜：删除或隐藏公开排行榜投影。
- 关闭分享链接：访问者立即失去只读入口。

删除状态必须优先于迟到上传，避免用户删除后旧请求又把森林重新创建。

## 11. 桌面与网页联动模型

### 11.1 推荐首期原则

> **桌面单一耐久写入者 + 网页富视觉镜像 + 双向瞬时触摸。**

在 M0/M1：

- 桌面负责真实 token、气泡收取、成长、阶段和本地存档。
- 网页负责展示森林、回顾、镜头和无经济影响的触摸。
- 云端保存带版本的私有快照和最小化领域事件。
- 网页不会直接设置 token、余额或果实数量。

这能快速创造跨端生命感，同时几乎消除经济与成长冲突。

### 11.2 目标逻辑链

~~~mermaid
flowchart LR
    A["Claude / Codex 本地日志"] --> B["桌面 Reader"]
    B --> C["桌面本地游戏状态"]
    C --> D["本地事件 Outbox"]
    D -->|"用户主动开启 Forest Sync"| E["私有事件与快照"]
    E --> F["网页个人森林"]
    F -->|"抚摸 / 吹风，短 TTL"| G["私有实时通道"]
    G --> H["桌面动画回应"]
~~~

重要边界：

- 云端事件是产品领域事件，不是 Claude / Codex 原始日志行。
- Realtime 只负责低延迟通知，不是事实源。
- 网页重连后按 revision 拉取状态，不能假设之前的实时消息一定收到。

### 11.3 建议事件

桌面到网页的耐久或半耐久事件：

- energy_collected
- growth_added
- tree_stage_changed
- tree_selected
- fruit_generated，M2 前只读
- snapshot_updated
- device_online
- sync_paused

网页到桌面的瞬时事件：

- tree_petted
- wind_triggered
- tree_focused
- ambient_ping

M1 不允许网页发送：

- set_tokens
- reset_growth
- set_balance
- harvest_fruit
- purchase_item
- modify_local_log_state

### 11.4 事件与快照

事件回答“发生了什么”，快照回答“森林现在是什么样”。

事件至少包含：

- event_id。
- forest_id。
- device_id。
- actor_id。
- entity_id。
- event_type。
- schema_version。
- client_occurred_at。
- server_received_at。
- payload。
- 幂等键。

快照至少包含：

- 当前森林状态。
- revision。
- schema_version。
- 已处理事件位置。
- 最后写入设备。
- 服务端更新时间。

原则：

- 事件上传失败后重试必须复用同一个 event_id。
- 服务端按 event_id 幂等处理。
- 客户端时间用于展示，服务端顺序用于收敛。
- 快照用于快速渲染，但不是唯一审计来源。
- 实时通道丢失不应破坏耐久状态。

### 11.5 状态权威与冲突策略

| 状态类型 | 首期权威 | 推荐冲突规则 |
|---|---|---|
| 原始日志、prompt、项目与路径 | 永远本地 | 不进入同步域 |
| token 与成长值 | 桌面领域事件 | 非负、幂等增量；不能被旧快照倒退 |
| 阶段与解锁 | 根据成长值重算 | 单调前进，旧状态不能重新锁定 |
| 树名 | 服务端 revision | 低风险最后有效写入胜出 |
| 装饰坐标 | M2 服务端 revision | 基于 base_revision 检测陈旧写入 |
| 果实、货币、购买 | M2 服务端事务 | 唯一果实／交易 ID，禁止整包覆盖 |
| 抚摸、吹风、摇树 | 瞬时实时事件 | 短 TTL，允许丢失，不影响经济 |
| 当前树与在线状态 | 按设备保存 | 一台设备不覆盖另一台设备界面 |
| 删除与撤销 | 服务端 tombstone | 优先于所有迟到上传 |

### 11.6 延迟与反馈要求

- 本端操作反馈：小于 100ms，不等待网络。
- 在线跨端回应目标：p95 小于 2 秒。
- 超过 3 秒：显示“连接较慢”，不要假装实时。
- 桌面离线时，网页触摸可在网页本地响应，但不应数小时后突然补播到桌面。
- 瞬时互动应有短 TTL，过期直接丢弃。

### 11.7 多设备策略

首期建议：

- 一个账号。
- 一台主要成长设备。
- 多个只读网页会话。

理由：

- 当前主树选择、气泡收取和本地账本都以单设备为中心。
- 两台桌面同时重扫日志可能产生重复成长。
- 多设备各自选择不同树会造成语义冲突。

后续支持多桌面时：

- 每台设备有独立 device_id、游标和事件序号。
- 配对时记录设备基线。
- 成长按设备级单调累计或稳定事件去重。
- 可考虑“工作电脑种苹果、家庭电脑种樱花”，将不同设备映射到不同林区，而不是争夺一个全局当前树。

## 12. 账号与设备配对

### 12.1 账号目标

网页森林需要的是可恢复用户身份，而不是复用当前单机匿名排行榜会话。

建议：

- Email Magic Link 或 OTP 作为 MVP。
- GitHub OAuth 作为后续可选入口。
- 支持将现有匿名排行榜身份认领或迁移到永久账号。
- 私人森林昵称与公开排行榜昵称可以不同。

Supabase 支持匿名身份链接永久身份，见 [Identity Linking](https://supabase.com/docs/guides/auth/auth-identity-linking)；无密码登录可参考 [Magic Link / OTP](https://supabase.com/docs/guides/auth/auth-email-passwordless)。

### 12.2 配对原则

配对不是把浏览器 refresh token 复制到桌面，而是：

1. 桌面发起一个短时设备请求。
2. 已登录网页确认该请求。
3. 服务端为桌面签发最小权限、设备专用、可撤销凭据。

配对要求：

- 配对码一次性使用。
- 短期有效。
- 有尝试次数限制。
- 桌面与网页同时显示目标账号和设备。
- 设备 ID 随机生成，不使用硬件指纹。
- 配对码和令牌不进入 URL、分析日志或客服工单。
- 提供设备列表、最后在线时间和撤销操作。
- 长期将敏感凭据存入 Windows Credential Manager 或 macOS Keychain 等系统凭据库。

### 12.3 身份迁移

现有 account.json 不能要求用户手工复制。迁移应：

- 在桌面端由受控流程发起。
- 明确是否保留现有排行榜行。
- 将匿名身份与永久账号关系交由服务端确认。
- 避免创建两个森林或重复总量。
- 失败时不影响本地树和原排行榜状态。

## 13. 隐私、信任与数据边界

### 13.1 品牌承诺的推荐措辞

> **Token Forest 的核心成长与数据分析继续在设备本地完成。账号和网络不是种树的前置条件；私人森林同步、公开排行榜和在线价格更新均是独立的可选联网能力。**

不能继续用笼统的“应用绝不联网”描述所有模式，因为产品已经存在用户主动开启的排行榜和可选联网能力。

### 13.2 三个独立授权

1. 私人 Forest Sync。
2. 公开排行榜。
3. 森林只读分享。

三者不能互相隐含：

- 开启私人同步不会自动上榜。
- 参加排行榜不会自动公开完整森林。
- 创建分享链接不会公开内部账号或设备信息。

### 13.3 数据分类

| 数据 | 本地保存 | 私人同步默认 | 公开默认 |
|---|---:|---:|---:|
| 原始 Claude / Codex 日志 | 是 | 否 | 否 |
| prompt、回答、代码 | 可能存在于原工具 | 否 | 否 |
| 会话标题、本地会话 ID | 是 | 否 | 否 |
| 项目、仓库、路径、Git 分支 | 是 | 否 | 否 |
| 模型、项目、会话明细 | 桌面 Dashboard | 否 | 否 |
| 成本估算 | 桌面 Dashboard | 否 | 否 |
| 树 ID、树种、成长值、阶段 | 是 | 用户开启后 | 分享设置决定 |
| 果实、解锁、库存、布局 | 是 | M2 开启后 | 默认否 |
| 领域事件 ID、revision、版本 | 是 | 是 | 否 |
| 粗粒度成长时间 | 是 | 可选最小化 | 默认隐藏 |
| 在线与最后同步状态 | 是 | 私人可见 | 默认隐藏 |

### 13.4 绝不为同步上传的内容

- 原始日志文件。
- prompt。
- AI 回复正文。
- 源代码。
- 会话标题。
- 项目和仓库名称。
- 本地文件路径。
- Git 分支。
- 精确会话活动时间线。
- 用于反作弊的原始工作内容。

即使不上传正文，精确活动时间也可能暴露作息。因此默认使用粗粒度时间，实时在线状态由用户决定是否显示。

### 13.5 用户可见的数据控制

设置页必须提供：

- 当前同步字段。
- 最近一次同步时间。
- 当前 revision 或可读的状态版本。
- 已配对设备。
- 当前公开信息。
- 暂停同步。
- 撤销设备。
- 下载个人数据。
- 删除云端森林。
- 关闭分享链接。
- 退出排行榜。

### 13.6 公开数据与内部 ID

公开森林应使用独立公开投影或公开 ID：

- 不直接暴露内部账号 ID。
- 不复用私人 Forest 表的公开 SELECT。
- 精确 token、在线状态和设备状态默认隐藏。
- 分享页只读取用户明确选择的字段。

当前排行榜数据库存在公开读取策略，网站 UI 不展示 user_id 不等于数据库字段技术上绝对不可访问。正式森林必须使用独立私有表和 RLS，不得沿用排行榜公共读取边界。

### 13.7 删除与保留

正式 Forest Sync 需要明确：

- 事件保留期限。
- 快照和备份保留期限。
- 删除任务状态。
- 反作弊或安全审计历史是否保留、为何保留、保留多久。
- 删除完成后的确认方式。

当前 Beta 的“关闭排行榜”可停止后续同步并请求删除公开行，但不应笼统描述为已确认删除所有后台历史。正式版需要可重试、可确认的删除任务和 tombstone。

## 14. 成长、奖励与经济设计

### 14.1 token 的正确角色

token 应承担：

- 真实历史记录。
- 主树成长输入。
- 长期阶段依据。

token 不应直接承担：

- 所有商店购买力。
- 稀有抽取次数。
- 唯一社交地位。
- 可以通过浪费调用刷取的高价值奖励。

### 14.2 推荐奖励结构

- **原始 token**：事实记录，不消费、不减少。
- **成长值**：推动树阶段，可对极端高量采用递减收益。
- **生态成熟度**：推动环境丰富，更关注长期与多样性。
- **果实／装饰货币**：来自里程碑、成熟和稳定积累。
- **居民与稀有事件**：由森林成熟度、树种组合、季节或长期陪伴触发。

### 14.3 递减收益原则

- 保证普通用户也能稳定看到成长。
- 极高用量继续计入历史，但游戏奖励不无限线性放大。
- 可以采用每日柔性上限，但不能让树停止记录真实 token。
- 不用“你今天还差多少 token”鼓励用户继续产生调用。
- 排行榜可以显示原始数据，但不让它控制全部生态与收藏。

### 14.4 无惩罚照料

- 浇水、抚摸和吹风是表达，不是生存条件。
- 不照料不会死亡。
- 不签到不会失去奖励。
- 果实不会在固定时间腐烂。
- 回归后不需要补做任务。
- 森林可以进入休息、沉睡或等待春天的状态，但不是负面惩罚。

### 14.5 交易边界

私人森林中的装饰和陪伴无需高强度反作弊。只有以下内容需要服务端权威：

- 可消费余额。
- 稀缺物品。
- 购买和赠送。
- 公开排名奖励。
- 未来可能涉及的交易。

不建议早期引入玩家交易或真钱价值。

## 15. 社交设计

### 15.1 社交目标

社交应提供：

> **有人来过的温度，而不是被比较和被掠夺的压力。**

### 15.2 推荐顺序

#### 第一阶段：只读访问

- 用户生成可撤销分享链接。
- 访客浏览森林和树木公开档案。
- 访客不能移动、收取或修改任何内容。

#### 第二阶段：善意痕迹

- 每日一次阳光。
- 给树浇水，但不改变核心数值。
- 留下短暂脚印。
- 发送森林明信片。
- 赠送低价值小种子或装饰。

#### 第三阶段：共同空间

- 两人共同培育一小片林地。
- 社区共同树或公益树。
- 共享里程碑。

### 15.3 不建议加入

- 偷果实。
- 破坏装饰。
- 强制互助。
- 默认公开在线状态。
- 访客实时跟踪。
- 以 token 消耗量决定全部地位。
- 没有审核能力前开放自由文本留言。

### 15.4 审核与滥用

在开放好友、文本、礼物或共享空间前，需要：

- 举报。
- 屏蔽。
- 限流。
- 礼物反刷。
- 名称和文本审核。
- 封禁与申诉。
- 未成年人和隐私规则评估。

因此 M0/M1 坚持私人森林最稳妥。

## 16. 反作弊边界

### 16.1 当前能力的准确描述

当前排行榜数据由客户端自报：

- RLS 能保证用户只能修改自己的行。
- 它不能证明本地 token 记录未被修改。
- 后台历史和 Ranger 可以辅助发现突增和明显异常。
- 管理员可以隐藏异常用户。
- 重新创建匿名身份仍可能绕过旧身份封禁。

因此当前更准确的描述是：

> **异常检测与人工管理，而不是不可伪造的服务端反作弊。**

### 16.2 正式森林最低校验

- JSON Schema 和字段长度校验。
- 树种枚举、阶段和整数范围校验。
- 服务端重算阶段和总量。
- 成长单调性。
- event_id 幂等。
- 突增速率和重放检查。
- metric_version 与合法迁移事件。
- 明确处理历史 v2 → v3 ×100 等合法迁移。

### 16.3 不让反作弊侵入私人空间

- 私人森林中的视觉成长不需要上传工作内容证明。
- 公开榜单应标注“设备自报”或验证等级。
- 稀缺奖励和交易需要更严校验。
- 不应为了榜单可信度上传 prompt、代码或日志。
- 如未来建立验证榜，应清楚说明额外数据和隐私代价。

## 17. 分期路线图

### 17.1 M0 · 私人 Forest Gallery

#### 目标

证明用户愿意登录网页查看自己的真实森林。

#### 范围

- 普通用户登录。
- 一次性设备配对。
- 私人 Forest 页面。
- 当前四棵主树。
- 四个基础林区。
- 树木档案。
- 最近同步时间和状态。
- 基础离线变化摘要。
- 拍照模式。
- 只读分享链接。
- 隐私和设备管理。

#### 暂不包含

- 实时双向互动。
- 商店、库存和装饰摆放。
- 好友访问。
- 网页收果。
- 多桌面合并。

#### 验收标准

- 用户能在一分钟内完成配对。
- 网页数据与桌面一致。
- 刷新后森林状态保持。
- 状态时间清楚可见。
- 用户可撤销设备、暂停同步和关闭分享。
- 移动端可查看并点击树。

### 17.2 M1 · Living Link

#### 目标

证明桌面和网页承载的是同一个生命。

#### 范围

- 桌面成长事件实时进入网页。
- 网页树冠、萤火和环境反馈。
- 网页抚摸、吹风触发桌面反应。
- 在线、连接较慢和离线状态。
- 离开期间变化摘要。
- 阶段升级动画。
- 事件幂等、断线恢复和过期瞬时事件丢弃。

#### 核心验收

- 桌面收气泡后，网页在目标延迟内响应。
- 网页摸树后，桌面产生明确但不打扰的回应。
- 重连不会重复增长。
- 过期触摸不会在数小时后补播。
- 用户能自然理解两端的关系。

### 17.3 M2 · 森林经营

#### 目标

让森林具有长期个性和布置价值。

#### 前置条件

- 独立树实例。
- 服务端事件与快照模型稳定。
- 果实和库存具有唯一 ID。
- 服务端交易与 revision 冲突处理完成。

#### 范围

- 树名、出生时间和坐标。
- 背包和装饰。
- 装饰摆放。
- 果实收取。
- 少量商店。
- 生态成熟度。
- 昼夜、天气和动物。
- 记忆树。

### 17.4 M3 · 活森林

#### 目标

让森林从“有树的场景”变成会持续发生故事的生态。

#### 范围

- 更丰富的动物居民。
- 林区恢复和小型建筑。
- 季节变化。
- 多设备独立林区，视验证决定。
- 里程碑纪念物。
- 长期成长回顾。

### 17.5 M4 · 温和社交

#### 目标

让森林之间形成轻量、善意的连接。

#### 范围

- 好友关系。
- 只读来访。
- 阳光、脚印和明信片。
- 小礼物。
- 共同林地。
- 公开权限细化。
- 完整审核、举报和封禁机制。

### 17.6 第一次正式发布的验收定义

第一次正式发布成功，不是因为网页上出现了四棵树，而是因为用户能完成：

1. 在桌面养出真实树木。
2. 登录并配对网页森林。
3. 在网页认出自己的树。
4. 看到真实成长历史。
5. 桌面收取气泡后，网页产生即时变化。
6. 网页抚摸树后，桌面产生回应。
7. 理解同步范围并能随时撤销设备。
8. 离开网页时，没有未完成任务和损失压力。

验收原则：

> **如果用户只看截图，它是一片好看的森林；如果用户同时打开桌面，它才会显露出 Token Forest 真正独特的生命感。**

## 19. 成功指标

### 19.1 北极星指标

推荐：

> **每周至少完成一次“桌面成长后回访网页森林并产生有效互动”的配对用户数。**

它比单纯页面访问量更能验证跨端闭环。

### 19.2 激活指标

- 首次树成长后点击网页森林入口的比例。
- 开始配对到完成配对的转化率。
- 配对用户首次成功看到真实树的比例。
- 首次森林会话中点击或抚摸树的比例。
- 首次体验跨端魔法时刻的比例。
- 从配对开始到看到真实树的中位时间。

### 19.3 留存指标

- 配对用户 7 日和 30 日森林回访率。
- 主树阶段升级后 24 小时内回访率。
- 每周发生一次轻互动的用户比例。
- 生成明信片或分享链接的比例。
- 长期离开后成功回归的比例。

### 19.4 体验质量指标

- 跨端事件 p50／p95 延迟。
- 同步失败率。
- 重复成长事件率。
- 配对失败率。
- 删除任务完成率。
- 页面加载时间。
- 场景平均帧率。
- 移动端交互成功率。
- 减少动画模式覆盖率和问题反馈。

### 19.5 情感验证

通过访谈或可选问卷验证：

- “我觉得这棵树是我养出来的。”
- “网页和桌面像是同一个生命。”
- “我愿意一个月后再来看它。”
- “森林让我放松，而不是增加任务。”
- “我理解哪些数据被同步到云端。”

### 19.6 护栏指标

不能只追求访问时长，还要观察：

- 是否诱导用户刻意浪费 token。
- 是否增加签到或收获焦虑。
- 桌面通知关闭率。
- Forest Sync 退出率。
- 用户对隐私范围的误解率。
- 网页森林的 CPU、GPU、内存和电量影响。
- 排行榜是否显著改变正常 AI 使用行为。

分析事件也应最小化，不采集树名、精确 token、项目或私有森林内容作为产品分析维度。

## 20. 用户研究与验证

### 20.1 概念验证

使用静态稿或可点击原型测试 5–8 名目标用户：

- 是否理解桌面树与网页森林的关系。
- 更偏爱四棵主树还是大量自动生成树。
- 最想进行的前三种互动。
- 是否愿意为森林注册和配对。
- 哪些同步字段令人担忧。
- 70% 陪伴、30% 经营是否符合预期。

### 20.2 场景原型

制作一个不接真实账号的森林原型：

- 横向拖动。
- 四个林区。
- 点击、聚焦和抚摸树。
- 模拟成长事件。
- 模拟桌面回应。
- 移动端基础交互。

观察：

- 用户是否无需教程就能发现互动。
- 视觉是否足够丰富但不混乱。
- 粒子和前景是否过度。
- 用户是否自然停留 1–3 分钟。
- 用户是否知道何时可以安心离开。

### 20.3 真实联动试验

小范围接入真实桌面数据：

- 验证配对。
- 验证延迟。
- 验证离线恢复和幂等。
- 验证状态时间是否可理解。
- 记录第一次看到两端回应时的反应。
- 访谈用户是否愿意长期保留同步。

### 20.4 核心假设

| 假设 | 验证方式 | 通过信号 |
|---|---|---|
| 用户愿意为真实森林配对 | 可点击原型 + 小范围实装 | 配对完成率和访谈意愿 |
| 四主树 + 生态足够像森林 | 场景原型对比 | 不要求大量重复树也感到丰富 |
| 跨端回应形成独特价值 | M1 原型 | 用户主动复现、表达惊喜 |
| 无惩罚仍能产生回访 | 4 周测试 | 回访不依赖红点和损失 |
| 私人默认提升信任 | 隐私文案测试 | 用户正确理解三种授权 |

## 21. 主要风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 森林只是短期新鲜感 | 回访快速下降 | 生态变化、记忆树、变化摘要和场所记忆 |
| 四棵树显得太稀疏 | 不像森林 | 用草木、动物、路径和地貌构建第二层生态 |
| 美术范围迅速膨胀 | 周期失控 | 首版复用正面树和现有装饰，限制四林区 |
| 高 token 获得高价值奖励 | 鼓励浪费 | 递减收益、里程碑奖励、柔性上限 |
| 网页变成第二个 Dashboard | 缺少情感差异 | 场景优先，数据按树展开 |
| 同步重复计算 | 成长、果实复制 | event_id、outbox、幂等、revision |
| 删除后被迟到上传恢复 | 破坏信任 | tombstone、撤销优先、可确认删除任务 |
| 用户担心日志泄露 | 不愿配对 | 最小字段、清楚同意、私人默认 |
| 动画影响性能 | 卡顿和耗电 | 粒子预算、分层暂停、减少动画 |
| 玩法过重 | 打断工作 | 30 秒完整访问、无每日任务 |
| 玩法过轻 | 长期动力不足 | 生态、装饰、动物和记忆系统分期 |
| 社交变成竞争 | 破坏气质 | 只读访问、善意互动、排行分离 |
| 公开榜单作弊 | 竞争失真 | 服务端校验、设备自报标识、人工审核 |
| 多设备冲突 | 数据不可信 | 首期单主设备，后续设备级事件与基线 |
| 桌面与网页视觉不一致 | 失去同一生命感 | 同一 tree_id、统一阶段和事件语义 |

## 22. 当前建议拍板的产品决策

| 决策 | 推荐方案 | 理由 |
|---|---|---|
| 森林由什么组成 | 四棵主树 + 自动生态 + 后续记忆树 | 保留情感对象，同时形成森林密度 |
| 产品气质 | 70% 安静陪伴，30% 轻量经营 | 符合开发者工作场景 |
| 视觉方向 | 横向 2.5D 像素森林 | 最大化复用当前资产 |
| 首期权威 | 一台桌面负责耐久成长 | 降低同步冲突 |
| 首期网页互动 | 抚摸、吹风等瞬时动作 | 快速制造跨端生命感 |
| 私密性 | 私人默认 | 与 local-first 品牌一致 |
| 排行榜 | 与私人森林分开 | 避免公开授权外溢 |
| 分享页 | 独立只读投影 | 控制公开字段 |
| 经济 | 不与原始 token 无限线性挂钩 | 避免鼓励浪费 |
| 惩罚 | 无死亡、无过期损失 | 降低焦虑和任务感 |
| 社交 | 只读和善意异步优先 | 降低审核与骚扰风险 |
| 首发范围 | M0 + M1 共同形成正式承诺 | 同时交付真实森林和跨端独特性 |

## 23. 尚待讨论的开放问题

- 四种树是否永远固定绑定某类 AI 来源？
- 一个用户未来是否允许拥有多棵同种主树？
- 主树成熟后，如何生成记忆树而不让用户感到“重置”？
- 用户是否能移动主树到其他林区？
- 季节由现实时间、用户选择还是成长状态决定？
- 树名和个性是否由用户填写，是否需要 AI 生成？
- 精确 token 是否继续适合作为公开排行榜主指标？
- 分享页显示哪些信息最安全且最有吸引力？
- 移动端只承担查看和轻互动，还是完整布置？
- 环境音是否原创，如何控制下载体积和授权？
- 多台桌面设备合并为一片森林，还是各自形成林区？
- M2 是否沿用当前果实与货币模型？
- 是否提供完全不登录的本地网页镜像模式？
- 是否允许用户导出可离线保存的森林快照或明信片？
- 公共社区树是否适合作为比排行榜更温和的社交中心？

## 24. 下一步建议

在进入代码前，建议按以下顺序推进：

1. **拍板本文件第 22 节的核心决策。**
2. **制作两张低保真流程图：**
   - 首次登录与设备配对。
   - 桌面收气泡与网页抚摸的跨端闭环。
3. **制作横向 2.5D 森林静态概念稿：**
   - 四个林区。
   - 桌面树在网页中的对应位置。
   - HUD 和树木档案。
4. **制作不接真实数据的 Living Link 交互原型。**
5. **对 5–8 名目标用户进行概念测试。**
6. **冻结 Forest Sync 数据字段、授权文案和删除规则。**
7. **编写技术设计：身份、设备、事件、快照、RLS、删除与迁移。**
8. **再进入 M0 / M1 实施。**

建议最先验证的单一场景是：

> **配对完成后，用户在桌面收取一个气泡，网页树随之生长；用户在网页摸一下树，桌面树轻轻回应。**

如果这一刻成立，Token Forest 就会从一个可爱的 token tracker，转变为一个用户愿意养几个月、几年，甚至不愿失去的数字森林。

## 附录 C：术语

| 术语 | 含义 |
|---|---|
| 主树 | 当前持续成长、桌面与网页共享身份的核心树 |
| 记忆树 | 保存一段成熟历史、不再承担实时成长的纪念树 |
| Forest Sync | 用户主动开启的私人森林同步能力 |
| 排行榜 | 独立授权的公开 token 排名 |
| 分享页 | 用户主动创建的只读森林公开投影 |
| 领域事件 | 树成长、阶段变化、抚摸等产品语义事件，不是原始 AI 日志 |
| 快照 | 用于快速加载的森林当前状态 |
| revision | 服务端用于判断状态版本和冲突的单调版本号 |
| outbox | 桌面断网时持久保存待上传事件的本地队列 |
| 瞬时事件 | 抚摸、吹风等可丢失、短 TTL、不影响经济的事件 |
